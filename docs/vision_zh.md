# 项目重构目标文档

## 1. 背景

这是一个已有的 C++ 编译器 / 代码生成项目。项目中存在较多历史包袱，包括：

- 自定义基础类型较多
- 资源管理方式偏旧
- 对 Windows 特定实现存在依赖
- 命名风格不统一
- 文档和注释不足
- 一些历史注释存在编码问题

本次重构的目标**不是推倒重写**，也**不是新建一个全新项目**，而是：

- 在原项目中渐进式重构
- 尽量保持现有构建链和主要行为可用
- 在不破坏主线的前提下，把项目逐步整理成更现代、更清晰、更易维护的 C++ 工程

## 2. 核心目标

### 2.1 原地渐进式重构

重构应在**现有仓库和现有工程结构中进行**。

优先原则：

- 初期保留当前项目布局
- 尽量维持现有构建目标可用
- 避免无必要的大拆大改
- 优先采用小批次、可验证的改动

这是一项分阶段的现代化整理工作，而不是一次性重写。

### 2.2 用标准 C++ 逐步替换历史基础设施

项目中大量自定义基础设施，应尽可能用标准库或标准风格替代，或至少先缩小依赖面。

重点对象包括：

- 自定义字符串类型
- 自定义容器，例如 `List`、`LinkedList`、字典类结构
- 自定义智能指针，例如 `refPtr`
- 自定义文件流 / 文本流等基础 I/O 抽象

优先替换方向包括：

- `std::string` / `std::wstring`
- `std::vector`
- `std::list`（仅在确有必要时使用）
- `std::unordered_map` / `std::map`
- 标准智能指针
- `<fstream>` 及相关标准 I/O 设施

重要约束：

- 不要求一次性替换所有公有接口
- 可以先替换内部实现
- 兼容层可以暂时保留，后续再逐步压缩暴露面

### 2.3 减少 Windows 专有依赖

长期目标之一，是减少项目对 Windows API 和平台专有行为的耦合。

这并不意味着当前阶段要立刻完成全平台迁移，而是意味着：

- 尽量不要继续引入新的 Windows 专属基础设施
- 在可行的地方优先使用标准 C++ 方案
- 为未来可能的跨平台改造提前清障

## 3. 代码质量目标

### 3.1 提升可读性与一致性

重构不只是“代码能跑”，还应明显提升代码可读性。

期望结果包括：

- 命名更统一
- 局部变量和函数名更清楚
- 历史缩写和含糊名字减少
- 实现更短、更直观
- 接口行为更一致

命名风格方向：

- 倾向统一的 C++ 风格
- 在合适的地方使用下划线风格命名
- 避免不同命名体系长期混杂

### 3.2 在现代化的同时修 correctness 问题

重构不能只是“换皮”，还应该顺手修掉真实问题。

重点包括：

- 边界条件错误
- 空输入 / 空字符串 / 空路径行为不一致
- 非法索引 / 非法区间处理不稳
- 未定义行为
- 脆弱的手工内存管理
- 隐含所有权问题
- 字符串 / 流 / 路径处理语义不一致

### 3.3 保持主构建链可用

本项目重构的重要原则之一是持续可验证。

每批改动都应尽量做到：

- 主构建保持可用
- 避免长期处于“改到一半编不过”的状态
- 做完一批就验证一批
- 优先采用低风险、可回退的推进方式

## 4. 注释与文档目标

这次重构不仅是技术清理，也是项目可理解性建设。

期望包括：

- 为关键模块补充更清晰的注释
- 增加文件级、模块级说明
- 对仍保留的兼容层解释其存在原因
- 对关键算法、关键 pass 给出简短说明
- 如果有好的外部资料，可补充参考链接或说明

### 4.1 历史注释恢复

项目中部分注释看似乱码，实际上可能是旧的 GBK 编码文本。

在可行的情况下，应尽量：

- 恢复原始注释含义
- 把文件逐步整理到 UTF-8
- 用更清楚的文字重写损坏或难懂的旧注释
- 尽量保留原作者留下的有效领域知识

## 5. 工程化目标

### 5.1 分阶段推进

项目应分阶段整理，而不是试图一次完成所有目标。

当前推荐顺序：

1. 稳住并现代化内部实现
2. 缩减自定义容器和手工资源管理的使用面
3. 增加最小回归测试护栏
4. 压缩兼容层暴露面
5. 最后再考虑更大范围的接口替换

### 5.2 增加最小回归测试

当前阶段不要求一次性搭出完整测试体系，但必须逐步降低验证成本。

初始测试重点包括：

- 字符串边界行为
- 路径和文件 I/O 边界行为
- 流和编码相关边界行为
- 建立一条轻量、可重复执行的最小自测路径

### 5.3 按专题成批推进

重构工作应尽量按专题组织，例如：

- `String` 与字符串语义
- `Stream` / `TextIO`
- 路径与文件辅助函数
- 局部容器标准库替换
- 模块内命名整理

这样可以减少上下文切换，提高验证效率。

## 6. 当前阶段明确不做的事

以下事项**不是当前阶段的优先目标**：

- 将 IR 整体改造成 LLVM 风格
- 重做 CFG 可视化工具
- 立刻完成整个项目的跨平台化
- 在项目尚未稳定前，先大规模重组整个仓库结构

这些方向未来可以再做，但不属于当前主线。

## 7. 期望的最终状态

最终希望这个项目能变成：

- 保留原有项目价值
- 大幅减少对自定义基础设施的依赖
- 在合理范围内更多使用标准 C++
- 风格更统一、命名更清晰
- 更容易理解和继续维护
- 修改风险更低
- 对历史脆弱点有最小测试护栏
- 为未来进一步架构升级或跨平台改造打下基础

## 8. 工作原则

当目标冲突时，优先级如下：

- 正确性优先于表面整理
- 渐进且可验证的进度优先于大而不稳的重写
- 内部实现现代化优先于一次性大规模接口破坏
- 清晰易懂优先于炫技式写法
- 标准库优先于重复造轮子，除非确有充分理由保留自定义实现
